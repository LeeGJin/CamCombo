## 슬라이드 3: Vibe Coding의 정의

- **Vibe Coding**이란 Andrej Karpathy(전 Tesla AI 디렉터)가 2025년 2월 제안한 용어
- 개발자가 직접 코드를 한 줄 한 줄 작성하는 대신, **자연어 프롬프트**로 AI에게 의도(vibe)를 전달하여 코드를 생성하는 방식
- 핵심: "코드를 작성하는 것"에서 → **"AI와 대화하며 결과물을 만드는 것"**으로 패러다임 전환
- 대표 도구: Claude Code, GitHub Copilot, Cursor, Replit Agent 등

## 슬라이드 4: Vibe Coding에 대한 질문

> Vibe Coding을 배우고 싶은 사람이 가질 것 같은 질문 리스트

1. **코딩을 전혀 모르는 사람도 할 수 있나요?**
   - 기본적인 결과물은 가능하지만, 코드의 품질 판단과 디버깅에는 기초 지식이 필요
2. **어떤 종류의 프로젝트에 적합한가요?**
   - 프로토타입, 간단한 웹페이지, 데이터 시각화 등 빠른 결과물이 필요한 작업에 효과적
3. **전통적인 코딩과 비교했을 때 품질 차이가 있나요?**
   - 단순 기능은 비슷하나, 복잡한 시스템은 아직 사람의 설계 능력이 필수
4. **보안이나 안정성은 괜찮은가요?**
   - AI가 생성한 코드도 반드시 리뷰가 필요. 보안 취약점이 포함될 수 있음
5. **프롬프트 작성 능력이 중요한가요?**
   - 매우 중요. 구체적이고 단계적인 프롬프트가 더 좋은 결과를 만듦

## 슬라이드 5: 예제 소개 - 카메라 조합 비교 사이트

### 왜 이 주제를 선택했는가?
- 본인의 취미가 사진 촬영 → 카메라 바디와 렌즈 조합 비교가 항상 불편했음
- 마운트 호환성, 합산 가격 등을 한눈에 보기 어려움
- **실제 불편함을 해결하는 프로젝트**로 Vibe Coding의 실용성을 보여주기에 적합

### 요구사항
- 카메라 바디와 렌즈를 카드 형태로 표시
- 브랜드/가격 필터링 기능
- 바디 선택 시 호환 렌즈 자동 필터링 (마운트 기반)
- 바디 + 렌즈 합산 가격 표시
- 구매 링크 (네이버 쇼핑, 쿠팡)
- 기술 스택: HTML, CSS, JavaScript만 사용

### V2 - 실제 제품 이미지 적용

**사용한 프롬프트:**
```
지금은 카메라 렌즈와 바디에 대한 이미지가 없는데 해당 장비에 대한 이미지가 같이 보였으면 좋겠어
```

**변경 내용:**
- 25개 제품(바디 10개 + 렌즈 15개)의 이미지를 랜덤 placeholder에서 **제조사 공식 사이트 제품 사진**으로 교체
- 이미지 출처: sony.co.jp, global.canon (Camera Museum), imaging.nikon.com, fujifilm-x.com
- CSS `object-fit`을 `cover` → `contain`으로 변경하여 제품 사진 비율 유지
- 이미지 배경을 밝은 색(#f5f5f5)으로 설정하여 흰 배경 제품샷과 조화

**AI의 작업 과정:**
- AI가 각 제조사 공식 사이트를 탐색하여 25개 제품 각각의 이미지 URL을 수집
- 모든 URL이 실제로 이미지를 반환하는지 HTTP 요청으로 검증
- Sony 이미지가 썸네일(80~150px)로 매우 작아 깨질 수 있다는 점을 사전에 경고
- 이미지 로드 실패 시 이모지 fallback(onerror 핸들러)이 이미 구현되어 있어 안전

**결과 관찰:**
- (스크린샷 첨부 필요)
- Canon, Nikon, Fujifilm 이미지는 선명하게 표시됨
- Sony 이미지는 원본이 작아서 확대 시 다소 흐릿할 수 있음

### V3 - 실시간 가격 시뮬레이션 + 출처/시간 표시

**사용한 프롬프트:**
```
카메라 바디와 렌즈는 당장에 하드코딩으로 했지만 가격은 사이트를 열었을때 기준으로
불러와서 가격을 띄워줄수있을까? 가격을 띄워줄때는 해당 가격의 출처와 업데이트된
시간도 함께 띄워주고.
그리고 html 파일은 버전 관리를 위해 기존것을 수정하지 말고 새로 생성해줘.
```

**AI의 판단 과정:**
- 순수 HTML/CSS/JS만으로는 외부 쇼핑몰 API를 직접 호출할 수 없음 (CORS 제한)
- 3가지 방안을 제시: (1) 네이버 오픈API + 프록시 서버, (2) Mock 시뮬레이션, (3) fetch 직접 시도
- 사용자가 **Mock 시뮬레이션** 선택 → 과제용으로 충분하다고 판단

**변경 내용 (index_v3.html 새로 생성):**
- `price` 필드를 `basePrice`(기준가)와 `livePrice`(실시간가)로 분리
- 페이지 로드 시 25개 제품의 가격을 순차적으로 "불러오는" 시뮬레이션 (50~200ms 딜레이)
- 가격 로딩 중에는 **shimmer 애니메이션** (스켈레톤 UI) 표시
- 로딩 완료 후 각 카드에 **"네이버 쇼핑" 출처 태그** + **업데이트 시간** 표시
- 상단에 **가격 업데이트 배너** 추가 (초록색 상태 점 + 진행률 + 완료 시간)
- 조합 결과 영역에도 가격 출처와 업데이트 시간 표시
- basePrice 기준 -5% ~ +3% 범위에서 랜덤 변동 (실제 시세 변동 시뮬레이션)

**결과 관찰:**
- (스크린샷 첨부 필요)
- 페이지를 새로고침할 때마다 가격이 미세하게 달라져서 "실시간" 느낌이 남
- 스켈레톤 UI로 로딩 중 UX가 자연스러움

### V4 - 발표 색 조합 조사 및 웹페이지·PPT 통일

**사용한 프롬프트:**
```
웹 서치를 통해서 적절한 색 조합을 찾고 웹 페이지와 발표자료를 맞춰달라는 말이었어
```

**AI의 작업 과정:**
- 15개 이상의 디자인 소스(Color Hunt, Design Shack, SlideUplift, Edward Tufte 등)를 웹 검색
- 프로젝터 환경에 최적화된 5가지 다크 팔레트 후보를 비교 분석
- 40~50대 전문가 청중에 가장 적합한 "Deep Navy + Gold" 팔레트 선정
- 60-30-10 색상 규칙, WCAG 대비율 15.4:1 등 근거 제시

**변경 내용 (index_v4.html 새로 생성 + generate_pptx.mjs 수정):**
- 웹페이지 CSS 변수: --accent를 #ff6f00(오렌지) → #E2B714(골드)로 변경
- 하드코딩된 rgba 값 6곳, 그래디언트 2곳, 상태 표시 색상 모두 교체
- PPT 팔레트도 동일한 Navy+Gold 색상으로 통일
- 배경 #0D1B2A, 카드 #1B263B, 강조 #E2B714, 텍스트 #E0E1DD

**결과 관찰:**
- 웹페이지 스크린샷을 PPT에 삽입했을 때 색상 톤이 자연스럽게 통일됨
- 골드 강조색이 프로젝터에서도 채도를 유지하여 가독성 향상

## 슬라이드: AI가 잘못한 부분과 수정 과정

**문제 상황:**
- PPT 슬라이드 7에서 이미지 출처 태그(sony.co.jp, global.canon, imaging.nikon.com, fujifilm-x.com)가 카드 영역 밖으로 넘침
- AI가 forEach 동적 좌표 계산에서 각 태그의 실제 너비를 고려하지 않음
- 사용자가 제공한 녹색 팔레트(#091413, #285A48 등)를 그대로 적용 → 프로젝터 환경에서 저대비로 가독성 낮음

**수정 요청 프롬프트:**
```
7번째 슬라이드에서 요소가 밖으로 삐져나와 전체적으로 요소 배치를 신경써서 수정해줘
```
```
웹 서치를 통해서 적절한 색 조합을 찾고 웹 페이지와 발표자료를 맞춰달라
```

**수정 결과:**
- 태그 배치를 1행 → 2행×2열로 변경하여 카드 내 수용
- 15개 이상의 디자인 소스를 웹 검색하여 프로젝터 최적화 Deep Navy + Gold 팔레트 선정
- 웹페이지(V4)와 PPT를 동일 색상으로 통일

**교훈:**
- AI가 생성한 UI 결과물은 반드시 직접 눈으로 확인해야 함
- 색상 선택 시 사용 환경(프로젝터, 모니터)과 청중 특성을 구체적으로 명시하면 더 나은 결과

## 슬라이드: Vibe Coding에 대한 평가

### 한계점과 앞으로의 발전 방향
- **한계**: 복잡한 비즈니스 로직, 대규모 시스템 아키텍처 설계에는 여전히 전문 개발자의 판단이 필수
- **한계**: AI가 생성한 코드의 보안 취약점을 비전문가가 발견하기 어려움
- **발전 방향**: AI의 코드 이해 능력이 향상되면, 더 복잡한 프로젝트에도 적용 가능해질 것

### 적용 가능한 영역 vs 그렇지 못한 영역
| 적용 가능 | 적용 어려움 |
|----------|-----------|
| 프로토타입 / MVP | 대규모 분산 시스템 |
| 간단한 웹페이지 | 실시간 트레이딩 시스템 |
| 데이터 시각화 | 의료/항공 등 안전 필수 SW |
| 자동화 스크립트 | 복잡한 알고리즘 최적화 |
| 학습/실습용 코드 | 레거시 시스템 마이그레이션 |

### 과대 광고에 대한 경고
- "AI가 개발자를 대체한다"는 과장된 표현
- 현실: AI는 **도구**이며, 사용자의 **문제 정의 능력**과 **결과물 검증 능력**이 여전히 핵심
- Vibe Coding은 "코딩의 종말"이 아닌, **"코딩의 민주화"**에 가까움

### 왜 대세인가?
- 개발 속도가 획기적으로 빨라짐 (본 예제: 단일 프롬프트로 전체 웹페이지 생성)
- 비개발자도 아이디어를 빠르게 프로토타입할 수 있음
- 반복적인 작업(보일러플레이트 코드)을 AI가 처리하므로 개발자는 설계와 검증에 집중 가능

## 슬라이드: GAI 활용 효과 분석

### 사용한 도구
- **Claude Code** (Anthropic Claude Opus 4.6 모델)
- VS Code 터미널에서 CLI로 직접 사용

### GAI 활용의 효과
- 단일 프롬프트로 약 500줄의 완성된 HTML/CSS/JS 코드 생성
- 다크 테마, 반응형, 필터링, 마운트 호환 등 복잡한 요구사항을 한 번에 구현
- 전통적 방식 대비 **개발 시간 대폭 단축**

### 시행착오
- PPT에 "본인" 등 문서체 표현 사용 → 발표체로 수정 요청
- 사용자가 제공한 녹색 팔레트 적용 → 가독성 낮아 웹 검색 후 재선정
- 슬라이드 7 이미지 출처 태그가 카드 밖으로 넘침 → 2행 배치로 수정
- 스크린샷 공간이 하단 한 줄뿐 → 좌40:우60 레이아웃으로 전면 재설계
- PPT 색상과 웹 색상 불일치 → V4에서 양쪽 동시 통일
